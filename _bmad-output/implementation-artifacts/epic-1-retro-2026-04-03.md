## Epic Review

### Summary

Epic 1 delivered the core product foundation needed for a trustworthy local-first ledger workflow:

- packaged `uv`-managed Python foundation
- install and readiness verification on supported macOS environments
- CI, deterministic smoke checks, and governed diagnostics artifacts
- interactive workspace initialization with a valid Beancount starter ledger
- skill-first structural review and approval workflow
- Codex-first quickstart guidance

Epic 1 completed 6 of 6 planned stories and established the trust model that later import and reconciliation work will depend on.

### What Went Well

- The team kept the repo aligned with the architecture instead of taking short-lived shortcuts. Story 1.1 created the packaged foundation cleanly, and later stories extended those boundaries rather than collapsing them.
- Trust and safety concerns were brought forward early. Story 1.3 added governed artifacts, structured workflow results, CI, and smoke checks before import complexity arrived.
- The product-vs-workspace boundary became much clearer over the epic. Story 1.4 introduced `workspace_template/` and a separate generated runtime workspace, which reduced future confusion about where durable user state belongs.
- The workflow moved decisively toward a skill-first interaction model. Story 1.5 reworked the earlier direction so authored behavior lives in `skill_sources/` and runtime skills are materialized during `auto-bean init`.
- Documentation stayed honest about scope. Story 1.6 positioned the current workflow as setup, validation, inspection, and structural review readiness rather than pretending the import pipeline already exists.

### Challenges and Friction

- Several stories required scope correction after implementation details became clearer. The biggest example was Story 1.5, which had to be reworked away from a CLI-led review flow and back toward the architecture's skill-first model.
- Trust-sensitive requirements had to be carried directly in story text because there was no UX artifact or `project-context.md`. That increased the burden on each story to restate boundaries and safety expectations.
- Story sequencing exposed dependency tension. Story 1.2 deliberately deferred workspace creation, Story 1.4 then had to introduce the runtime repo and template, and Story 1.5 had to retrofit authored skill installation into that new boundary.
- Reviews surfaced important fail-closed gaps after initial implementation, especially around partial workspace creation, Git initialization, PATH assumptions, and structured error handling. The team responded well, but the pattern suggests we should assume trust-boundary bugs will appear late unless we test for them explicitly.

### Recurring Patterns

- Boundary clarification was a repeated theme: product repo vs generated workspace, authored skills vs installed runtime skills, CLI helpers vs skill-led workflows, and governed artifacts vs canonical ledger files.
- Deterministic verification kept paying off. Once CI, smoke checks, and structured artifacts were in place, later stories had a more stable place to anchor behavior changes.
- The team improved by tightening scope when needed instead of defending the first implementation. Stories 1.2, 1.5, and 1.6 all show course correction toward a narrower, more architecture-aligned result.

### Technical Debt and Risks

- The import pipeline does not exist yet, so the current quickstart ends at a trustworthy baseline rather than first imported value.
- Epic 1 created several important workflow contracts that Epic 2 must reuse consistently: structured artifacts, fail-closed validation, git-backed review visibility, and skill-first user guidance.
- Because trust-sensitive behavior currently lives mostly in stories, docs, and tests, there is still some risk of drift unless shared guidance continues to be centralized in `skill_sources/`, workspace docs, and stable workflow helpers.

### Key Lessons

1. Trust boundaries need explicit artifacts, not just good intentions. CI, diagnostics, validation, and review traces should be treated as product features.
2. Repo boundary decisions are foundational. Separating authored product assets from generated workspace state prevented later stories from mixing runtime behavior into the product repo.
3. Skill-first architecture needs to be enforced early. When a workflow drifts toward CLI-first convenience, later correction is more expensive.
4. Honest documentation improves product trust. Stating what is not yet supported is better than smoothing over gaps.
5. Review findings are a primary design input for this product. Fail-closed bugs should be expected and covered with explicit regression tests.
